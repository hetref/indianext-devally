PROBLEM STATEMENT

Design and develop a smart cyber defense platform that can detect, analyze, and explain emerging cyber threats using AI/ML techniques.
The system should be capable of identifying one or more of the following threat areas:

—	Phishing emails or phishing messages
—	Malicious or suspicious URLs
—	Deepfake audio or video impersonation indicators
—	Prompt injection or manipulated AI inputs
—	Anomalous user behavior or suspicious login patterns
—	AI-generated malicious content or deceptive communication

The solution must not only detect threats, but also provide clear explanations for why a particular input was marked suspicious. The system should assist users, administrators, or security teams in understanding the threat, assessing its severity, and receiving suggestions for mitigation or safe action.

BACKGROUND & CONTEXT

Cyber attackers are increasingly using AI tools to create more believable scams, evade detection, and automate attacks. Organizations need intelligent systems that not only detect such attacks but also justify their decisions.

If a system flags something as malicious, the user should know:

—	What made it suspicious
—	What evidence was used
—	How confident the system is
—	What action should be taken next

This makes the challenge highly relevant to modern cybersecurity needs and future-ready digital infrastructure.

WHAT WE EXPECT FROM YOU

Your prototype must demonstrate the following core components. These are not optional — every team is expected to address all of them:

1.  Threat Input Module
Your system must accept one or more forms of real input — such as a message, email, URL, activity log, or prompt text. The input must be meaningful and relevant to a real cyber threat scenario.

2.  Detection Module
Your system must use an intelligent method to identify suspicious or malicious patterns in the input. The detection logic must go beyond a simple yes/no output — it should reflect understanding of the threat.

3.  Explainability Module
Your system must explain its prediction. It is not enough for the system to say something is dangerous — it must show the user why. Judges will specifically evaluate the quality and clarity of your explanations.

4.  Response or Recommendation Module
Your system must suggest a next step or action based on what it detected. This should be practical and relevant — not generic.

5.  User Interface / Dashboard
Your prototype must have a visible interface that shows the input, the result, the risk level, the explanation, and the suggested action clearly. Judges will interact with your system.
